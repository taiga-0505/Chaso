# C++ Build & Log Setup Workflow（ビルド＆ログ環境構築ワークフロー）

> **概要**: Visual Studio等に依存せず、CLI（ターミナル）から「ビルド → 実行」を一発で完結させる PowerShell スクリプト（`build_and_run.ps1`）を構築し、ユーザーの好みに合わせた独自のログ管理構造を導入するエージェント用ワークフロー。過去の会話や前提コンテキストがゼロの別プロジェクトでも安全・確実に行える。

---

## Phase 1: スキャンとヒアリング（現状把握）

**Goal**: プロジェクトの既存のビルドシステム、出力先ディレクトリを特定し、ユーザーが望むログの出力・整理構成を決定する。

### Step 1.1: スキャン（環境の特定）

事前の質問なしに、まずはディレクトリをスキャンして以下の情報を特定する。
1. **ソリューションファイル (`.sln`) またはビルド定義ファイル (`CMakeLists.txt` 等) の位置**
2. **生成される実行ファイル (`.exe`) の出力先**
3. **実行時の作業ディレクトリの推定**（テクスチャなどアセットが相対パスで呼ばれるディレクトリ）
4. **既存のログシステム (`Log.cpp`, `Logger.h` 等) の有無と実装**

### Step 1.2: ログ構成とファイル配置のヒアリング（★必須★）

スキャン結果を元に、ユーザーに構成の好みをヒアリングする。
**注意**: ログのフォルダ構成には個人のこだわりが出るため、必ず以下の選択肢（または自由記述）を提示すること。

```markdown
🚀 ビルド＆実行の自動化フローを構築します。
その前に、ログファイルとスクリプトの配置についてご希望を教えてください。

【1. ログフォルダの構成】
A. ビルドとアプリで分ける（例: `logs/build/` と `logs/app/`）
B. ひとつのフォルダにまとめる（例: `logs/` のみ）
C. その他（お好みがあれば教えてください）

【2. スクリプトの配置】
A. プロジェクト直下を汚さず `scripts/` の中に隠す
B. プロジェクト直下にポン置きする

指定がなければ「1:A、2:A」のおすすめ構成で進めます！
```

## Phase 2: 診断（問題分析と要件定義）

**Goal**: 現在のC++のログ出力システムの問題（ファイル出力がない、バッファリングで止まる等）を診断し、スクリプトに必要なパス情報を整理する。

### Step 2.1: C++側ログシステムの問題診断

既存のC++のコード（`Log.cpp` 等）を `view_file` または `grep_search` で調査し、以下を確認する。
- ファイルへ出力（`std::ofstream` 等）しているか？
- アプリ初期化のたびに上書き（`trunc`）モードで過去ログを消していないか？
- 出力時に `std::endl` や `flush` を行っているか？（バッファリングによりアプリ実行中にログが書き出されない問題を防ぐため）

### Step 2.2: `.gitignore` の影響調査

新しく生成するフォルダ（`logs/`, `scripts/`）が `.gitignore` に含まれているか確認し、後で追記するリストを作成する。

---

## Phase 3: 構築・再構成（実行）

**Goal**: ヒアリング結果に従い、ターミナルと VS Code タスクで使える `build_and_run.ps1` を作成し、C++ログコードを改修する。（※ `run.bat` などのバッチファイルは作成しない）

### Step 3.1: build_and_run.ps1 の作成

以下の要素を完全に満たす PowerShell スクリプト（`build_and_run.ps1`）を作成する。

1. **構成のパラメータ化**:
   `param([ValidateSet("Debug", "Development", "Release")] [string]$Configuration = "Debug")`
2. **ツールの自動検出**:
   `vswhere.exe` などを利用して `MSBuild.exe` のパスを自動解決する。
3. **パスの絶対解決**:
   配置場所（`scripts/` か直下か）に依存せず、`$PSScriptRoot` を利用してプロジェクトのルートと出力先を正しく解決する。
4. **ビルド時のログ保存**:
   ヒアリングで決定した `logs/build/` 等のディレクトリに、タイムスタンプ付きで詳細なビルドログを分岐して保存する。
5. **例外処理**:
   ビルド失敗時は即終了し、アプリを立ち上げないこと。

### Step 3.2: C++ログシステムの改修

Phase 2.1 の診断とユーザーの希望に基づき、ログコードを修正する。
1. アプリケーションログの出力先を決定したフォルダ（例: `logs/app/`）に向ける。
2. 二重の `Initialize()` によるバグがないか確認し、ファイルストリームを正しく生成させる。
3. 即座にログが書き出されるよう、ファイル出力部に `flush()` や `std::endl` を組み込む。

### Step 3.3: VS Code でのワンクリック化（tasks.json）

ターミナルで長いコマンドを打たなくて済むように、`.vscode/tasks.json` を作成・更新する。
- 種類: `type: "shell"`
- コマンド: `powershell -ExecutionPolicy Bypass -File "./scripts/build_and_run.ps1" -Configuration Debug` （配置に合わせてパスを変更）
- ショートカット（Ctrl+Shift+B）で動作するように `"group": { "kind": "build", "isDefault": true }` を設定する。

### Step 3.4: 掃除と Git 設定

1. `logs/` や作成した `scripts/` などを `.gitignore` に追記する。
2. 過去の散乱したログファイルがあれば、PowerShell の `Move-Item` を使用して決定したディレクトリ構造に合わせて移動整理する。

---

## Phase 4: 検証

**Goal**: スクリプトが単独で正しく動作し、タスクから正常に呼び出せるかを検証する。

### Step 4.1: スクリプトの動作テスト

```powershell
powershell -ExecutionPolicy Bypass -File "scripts/build_and_run.ps1" -Configuration Debug -BuildOnly
```
を実行し、ビルドが成功すること、指定した場所にビルドログが生成されていることを確認する。

### Step 4.2: ログディレクトリの確認

`logs/` ディレクトリ配下に出力されたアプリとビルドのログファイルが存在し、中身が書き込まれていることを確認する。

---

## Phase 5: 報告

**Goal**: ユーザーに完了報告と使い方を伝える。

### Step 5.1: ウォークスルー (.md) の生成

アーティファクトとして手順や構成をまとめた `.md` ファイルを提示する。
必須報告事項：
1. **生成したディレクトリ構成のツリー図**
2. **VS Code のショートカット（Ctrl+Shift+B）で実行できる旨の説明**
3. **ターミナルから手動で呼び出す際のコマンド**（引数の指定方法も含む）

---

## 異常系ハンドリング

| 状況 | 対応 |
|:-----|:-----|
| MSBuild等が見つからない | ユーザーにVisual Studioのインストール有無やパスを確認する |
| アプリ実行中にログがファイルに記録されない | C++の出力部に `std::flush` や `std::endl` が欠けていないか再調査する |
| 構造が深すぎて実行できない | `$PSScriptRoot` を使って相対から絶対パスを生成できているかチェックする |
