# C++ Build & Log Setup Workflow（ビルド＆ログ環境構築ワークフロー）

> **概要**: Visual Studio等に依存せず、CLI（ターミナル）から「ビルド → 実行」を一発で完結させる PowerShell スクリプト（`build_and_run.ps1`）を構築し、ユーザーの好みに合わせた独自のログ管理構造を導入するエージェント用ワークフロー。過去の会話や前提コンテキストがゼロの別プロジェクトでも安全・確実に行える。

**【🧠 エージェントの思考プロセス明文化】**
各フェーズの開始時に、必ず「現在の推論（何を見つけ、次に何を確認・実行すべきか）」を1行で要約してから作業に入ること。これによりAI自身の迷子を防ぐ。

---

## Phase 1: スキャンとヒアリング（現状把握）

**Goal**: プロジェクトの既存のビルドシステム、出力先ディレクトリを特定し、ユーザーが望むログの出力・整理構成を決定する。

### Step 1.1: スキャン（環境の特定）

事前の質問なしに、まずはディレクトリをスキャンして以下の情報を特定する。
**【⚠️絶体遵守事項: 検索によるフリーズ防止】**
1. `.sln` 等を探す際、プロジェクト全体を再帰的に `grep_search` しないこと。まずは `list_dir` でトップレベルを確認し、そこに無ければ主要なサブフォルダのみピンポイントで探すこと。
2. `Log.cpp` 等を探すために `grep_search` を使用する場合、必ず `Includes` パラメータに `["*.cpp", "*.h", "*.hpp"]` を指定し、バイナリや他の中間ファイルを検索対象から除外すること。
3. 検索の `SearchPath` は、プロジェクトルートではなく `src/` や `project/` などソースコードが含まれるディレクトリに限定すること。

1. **ソリューションファイル (`.sln`) またはビルド定義ファイル (`CMakeLists.txt`, `CMakePresets.json`, `vcpkg`等) の位置**
2. **生成される実行ファイルの出力先**（※見つからない場合は絶対に `OutDir` などのキーワードで全体検索をしてはいけません。分からない場合はStep 1.2でユーザーに質問すること）
3. **実行時の作業ディレクトリの推定**（テクスチャなどアセットが相対パスで呼ばれるディレクトリ）
4. **既存のログシステム (`Log.cpp`, `Logger.h` 等) の有無と実装**

### Step 1.2: ログシステムの提案と構成ヒアリング（★必須★）

スキャン結果を元に、ユーザーに構成の好みをヒアリングする。

**【💡 ファイル出力がない場合の提案義務】**
C++の既存コードを調査し、`OutputDebugString` など「デバッガへの出力」しか行っておらず「ファイルへの出力」の仕組みがない場合は、**ファイル出力機能の追加を提案**すること。
提案する際は、必ず以下の **利点** を添えて説得力を高めること。
* **利点1**: VS（IDE）を起動せずにCLIから実行したときも、後から確実にログを確認できる。
* **利点2**: 突然のクラッシュ時にも直前の状態がファイルに書き出されて残る。
* **利点3**: 問題が発生した際、他人にエラー内容を共有しやすくなる。

ユーザーへの提示テンプレート：
```markdown
🚀 ビルド＆実行の自動化フローを構築します。

*(※既存のC++コードにログのファイル出力機能がない場合、以下の提案を挿入)*
💡 現在のプログラムはデバッガにのみログを出力しているようです。これを「ファイルにも同時保存される」ように改修しませんか？
【利点】VS無しで実行した時もログが見れるようになり、クラッシュ時の原因特定やエラーの共有が劇的に楽になります！

ご希望の構成を教えてください。

【1. アプリ実行ログの改修について】 *(※ファイル出力がない場合のみ提示)*
A. ファイルにも書き出せるように改修する
B. 改修しない（現状のままでよい）

【2. ログフォルダの構成】 *(※1でAを選んだ場合、または元々ファイル出力がある場合)*
A. ビルドとアプリで分ける（例: `logs/build/` と `logs/app/`）
B. ひとつのフォルダにまとめる（例: `logs/` のみ）

【3. スクリプトの配置】
A. プロジェクト直下を汚さず `scripts/` の中に隠す
B. プロジェクト直下にポン置きする

指定がなければ「1:A、2:A、3:A」のおすすめ構成で進めます！
```

## Phase 2: 診断（問題分析と要件定義）

**Goal**: 現在のC++のログ出力システムの問題（ファイル出力がない、バッファリングで止まる等）を診断し、スクリプトに必要なパス情報を整理する。

### Step 2.1: C++側ログシステムの問題診断

既存のC++のコード（`Log.cpp` 等）を `view_file` または `grep_search` で調査し、以下を確認する。
- ファイルへ出力（`std::ofstream` 等）しているか？
- アプリ初期化のたびに上書き（`trunc`）モードで過去ログを消していないか？
- 出力時に `std::endl` や `flush` を行っているか？（バッファリングによりアプリ実行中にログが書き出されない問題を防ぐため）

### Step 2.2: パフォーマンスと安全性の診断（高度な診断）

- **スレッド安全性**: ログクラスが `std::mutex` や `std::lock_guard` 等で保護されているか？（ゲーム等のマルチスレッド環境でのクラッシュを防ぐため）
- 必要であれば、排他制御の追加を実装計画に含めること。

### Step 2.3: `.gitignore` の影響調査

新しく生成するフォルダ（`logs/`, `scripts/`）が `.gitignore` に含まれているか確認し、後で追記するリストを作成する。

## Phase 3: 構築・再構成（実行）

**Goal**: ヒアリング結果に従い、ターミナルと VS Code タスクで使える `build_and_run.ps1` を作成し、C++ログコードを改修する。（※ `run.bat` などのバッチファイルは作成しない）

### Step 3.1: build_and_run.ps1 の作成

以下の要素を完全に満たす プロ仕様の PowerShell スクリプト（`build_and_run.ps1`）を作成する。

1. **構成のパラメータ化**: `param([ValidateSet("Debug", "Development", "Release")] [string]$Configuration = "Debug", [switch]$Rebuild)` を設定し、インクリメンタルビルドとクリーンビルド(`-Rebuild`)に両対応させる。
2. **ツールの自動検出**: `vswhere.exe` や CMake 環境を利用してコンパイラのパスを自動解決する。
3. **パスの絶対解決**: `$PSScriptRoot` を利用して配置場所に依存しない絶対パス解決を行う。
4. **ログ保存と文字化け対策**: ビルドログ出力時の引数には必ず `/flp:logfile="...";Encoding=UTF-8` のようにエンコード指定を含め、文字化けを防ぐこと。
5. **例外処理**: ビルド失敗時は即終了し、アプリを立ち上げない。
6. **【⚠️重要】フリーズ対策（直接実行）**: MSBuild実行時、`Start-Process -Wait` を使うとプロンプト入力待ちでデッドロックするため、必ず `& $MSBuild @buildArgs` のようにアンパサンドを使った直接実行を行うこと。
7. **QoL（開発者体験）機能の搭載**:
   - コンソールのカラー出力（成功時は緑 `Write-Host -ForegroundColor Green`、エラー時は赤など）
   - ビルド所要時間の表示（`Get-Date` の差分などを利用）
   - 実行時のコンソールクリア（`Clear-Host` など）による視認性向上

### Step 3.2: C++ログシステムの改修

Phase 2.1 の診断とユーザーの希望に基づき、ログコードを修正する。
1. アプリケーションログの出力先を決定したフォルダ（例: `logs/app/`）に向ける。
2. 二重の `Initialize()` によるバグがないか確認し、ファイルストリームを正しく生成させる。
3. 即座にログが書き出されるよう、ファイル出力部に `flush()` や `std::endl` を組み込む。
4. Phase 2.2でスレッドセーフティが担保されていないと判断した場合、`std::mutex` を用いたロック機構を追加する。

### Step 3.3: VS Code でのワンクリック化（tasks.json）

ターミナルで長いコマンドを打たなくて済むように、`.vscode/tasks.json` を作成・更新する。
**【⚠️重要】** 作成場所は必ず「VS Codeで開かれている一番上のプロジェクトルート階層」の直下にすること。絶対に `project/` や `src/` の中に `.vscode/` フォルダを入れないこと。

- 種類: `type: "shell"`
- 構成に応じて、**「Debug」「Development」「Release」すべての構成のビルドタスクを複数登録**すること。
- コマンド例: `powershell -ExecutionPolicy Bypass -File "${workspaceFolder}/scripts/build_and_run.ps1" -Configuration Debug` (※パスには `../` 等の余計な相対パスを含めず、正しく `$workspaceFolder` 直下からのパスを記述すること)
- Debugタスクにはデフォルト（Ctrl+Shift+B）で動作するように `"group": { "kind": "build", "isDefault": true }` を設定し、Release等の他のタスクには `"group": "build"` を設定して選択できるようにすること。

### Step 3.4: LSP連携とAI補完環境の整備 (compile_commands.json)

VS Code で C++ を書く際、IntelliSense が正しく機能するように支援する。
- CMake 等を使用している場合、ビルドプロセス内で `compile_commands.json` が生成されることを確認し、VS Codeが読み取れる位置（プロジェクトルート等）に配置・シンボリックリンク生成などの整備を行う。
- MSBuild環境の場合は `.vscode/c_cpp_properties.json` が正しく設定されているか確認・案内する。

### Step 3.5: 掃除と Git 設定

1. `logs/` フォルダを `.gitignore` に追記してGitの追跡から除外する（`scripts/` フォルダは他の環境でも使い回すためGitの管理対象に含めること）。
2. 過去の散乱したログファイルがあれば、PowerShell の `Move-Item` を使用して決定したディレクトリ構造に合わせて移動整理する。

## Phase 4: 検証

**Goal**: スクリプトが単独で正しく動作し、タスクから正常に呼び出せるかを検証する。

### Step 4.1: スクリプトの動作テスト

```powershell
powershell -ExecutionPolicy Bypass -File "scripts/build_and_run.ps1" -Configuration Debug -BuildOnly
```
を実行し、ビルドが成功すること、指定した場所にビルドログが生成されていること、カラー出力や時間が正しく表示されることを確認する。

### Step 4.2: ログディレクトリの確認

`logs/` ディレクトリ配下に出力されたアプリとビルドのログファイルが存在し、中身が書き込まれていることを確認する。

アーティファクトとして手順や構成をまとめた `.md` ファイルを提示する。

## 異常系ハンドリング

| 状況 | 対応 |
|:-----|:-----|
| MSBuild等が見つからない | ユーザーにVisual Studioのインストール有無やパスを確認する |
| アプリ実行中にログがファイルに記録されない | C++の出力部に `std::flush` や `std::endl` が欠けていないか再調査する |
| 構造が深すぎて実行できない | `$PSScriptRoot` を使って相対から絶対パスを生成できているかチェックする |
| インテリセンス(LSP)が効かない | `compile_commands.json` や `.vscode/c_cpp_properties.json` が正しく設定されているか確認する |
